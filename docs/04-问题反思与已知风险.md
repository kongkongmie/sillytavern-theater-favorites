# 问题反思与已知风险

## 这次开发中出现过的问题

### 1. 先做了功能，后补安装体验

最初仓库只有前端和后端源码，没有自动安装器，导致说明要求新手手动复制目录、修改 `config.yaml`。这不是“文档写复杂了”，而是产品缺少安装能力。以后涉及前后端双组件时，安装器必须和首版一起设计。

### 2. 设置模型重复且反直觉

早期每个工具同时出现“启用”和“显示快捷按钮”，状态容易不一致。小剧场收藏夹也曾把识别、显示和动作混在一起。以后一个开关只表达一个结果，用户能直接观察到开关效果。

### 3. 预览高度反复修补

HTML 小剧场高度差异极大。固定高度会截断，完全跟随内容又可能因 ResizeObserver、MutationObserver 和滚动条变化形成反馈循环，导致背景无限延长。正确方向是：高度有合理上下限、外层负责滚动、iframe 尺寸更新要去重和限频，不能盲目信任每次上报值。

### 4. 为了“还原”而套太多容器

多层卡片、边框和 details 会挤压原小剧场宽度，破坏响应式布局和交互。管理页应只保留标题行、操作区和一个直接预览区域，不要把原内容塞进层层装饰框。

### 5. 收藏识别不能只绑定消息级按钮

一条回复可能有多个小剧场。收藏按钮必须绑定具体候选块，并靠近该块顶部；去重必须使用完整内容哈希，刷新后也要从后端状态恢复“已收藏”。

### 6. 第三方兼容不能猜 DOM

拟界文库来自酒馆助手注入，可能跨 iframe、Shadow DOM 或动态重建。仅靠固定选择器会时好时坏，还可能误改它的样式。兼容层应隔离、可关闭、只在目标界面存在时显示按钮，并尽量读取第三方公开状态或稳定数据，而不是大范围扫描和改样式。

### 7. “完整保存”必须和保护上限同时说明

当前后端 `MAX_FIELD_LENGTH` 为 2,000,000 个字符。普通小剧场通常远低于此值，但极端超长字段仍会被截断。下一版应改为按请求体字节大小明确拒绝并提示，或提高/取消字段截断，绝不能静默截断后仍声称完整保存。

### 8. 固定弹窗里不能直接对条目使用 `scrollIntoView()`

`overflow: hidden` 的固定弹窗外壳仍可能被浏览器当作可程序滚动容器。对长条目调用 `scrollIntoView()` 时，不只列表会滚，弹窗外壳也可能产生隐藏的 `scrollTop`；条目删除或收起后，标题和顶部按钮便会被裁掉。以后只能计算条目与列表的相对位置并修改列表自身的 `scrollTop`，同时让弹窗外壳使用 `overflow: clip` 并保持滚动位置为零。

### 9. 手机阅读页要先给正文分配空间

不能把桌面上的标题、状态、按钮、说明和全部筛选框原样纵向堆到手机上，否则真正的预览会被推到半屏以下。手机端应保留高频搜索和明确的入口，把低频筛选与单条管理操作折叠起来；同时减少左右留白，并用视口高度而不是固定小像素值决定交互式预览的最低高度。

## 当前已知风险

### 高优先级

- 极端超长字段可能被 `MAX_FIELD_LENGTH` 静默截断。
- HTML 小剧场可以运行收藏内容自带的脚本；sandbox 降低风险但不是绝对隔离。
- 拟界文库升级 DOM 或内部数据结构后，兼容识别可能失效。

### 中优先级

- iframe 高度算法需要继续覆盖超长、动态加载、内部滚动和交互式页面。
- 本地收藏达到数百 MB 时，导出完整 JSON 会产生较大内存峰值。
- 导入大备份目前是整包解析，手机和低内存服务器可能卡顿。
- 每条最多 20 个标签；这是单条限制，不是全局标签种类限制。

### 低优先级

- 当前仅中文界面。
- `manifest.json` 作者字段为 `Codex`，前端标题显示 `@KKM`；发布前可再次确认署名口径。
- KKM Tools 的集合入口仍属于可选后续工作，独立插件不依赖它。

## 后续建议顺序

1. 修复超长内容的静默截断，增加清晰的大小提示和失败反馈。
2. 为保存、读取、导入导出和索引重建补自动测试。
3. 用多个真实 HTML 小剧场建立回归样本，专门测试 iframe 高度与交互。
4. 把拟界文库适配封装成独立模块，并增加版本失配提示。
5. 再考虑多语言、批量标签和更大数据量下的流式导入导出。
